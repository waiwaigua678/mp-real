# Stage 9 硬件验证：移动到录制机器人状态

这是操作员清单，不是自动运动测试授权。Piper 与 RM2 的每个门禁必须独立执行；一台
机器人通过不代表另一台通过。

## 前置条件

- 记录 Git commit、机器人、操作员、目标数据集/episode/sample、映射配置指纹、
  ActionSpec 快照和计划 hash。
- 清理工作区，验证物理停止流程，姿态速度不得高于已配置的保守值。
- 确认厂商停止、健康、关节限位和工作区验证可用。当前实现缺任一能力时会拒绝真机
  移动；不得绕过。
- 不得把 action、raw action、expert action 或录制图像作为目标状态或实时部署观测。

| 门禁 | 操作 | 继续前必须满足 |
| --- | --- | --- |
| 0 | 不带 `--execute` 运行 `mp-move-to-recorded-state`，检查模式、单位、目标值和映射报告。 | 不创建 Robot；目标来自 `observation.state`；报告有效。 |
| 1 | 仅使用 Web **连接机器人用于姿态移动**。 | 可读机器人反馈；不启用、不复位、不连相机/策略、不运动。 |
| 2 | 选择与实时姿态近似相同的样本，生成实时计划但不执行。 | 当前状态/增量/单位/计划 hash 可见且符合预期。 |
| 3 | 工作区清空后，以低速执行最小关节增量。 | 进度在跟踪限制内；物理与 Web 停止均安全中止。 |
| 4 | 不交接策略地移动到附近录制状态。 | 最终为 `reached`（或经明确审查的警告）。 |
| 5 | 到达后选择 **连接相机和 Policy**。 | 不复位；丢弃预热动作；控制循环未运行。 |
| 6 | 以 `max_steps=1` 选择 **二次确认开始推理**。 | 首个执行动作来自新鲜真实相机/状态观测，而非录制帧。 |
| 7 | 运行简短低速部署。 | 录制元数据含源 sample、`move_plan_id` 和目标跟踪误差。 |

发生意外运动、健康失败、跟踪误差越界、控制超时、通信错误或陈旧计划拒绝时：先物理
急停；再停止姿态/部署控制器并等待 worker join；断开连接、保留日志和 `.inprogress`
录制，记录计划 hash 与原始错误；原因审查前不得进入后续门禁。
